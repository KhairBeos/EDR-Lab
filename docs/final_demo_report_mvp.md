# Final Demo Report MVP

## Purpose

Phase 7 adds a local operator-facing final demo report for the EDR platform. After Phase 9 and Phase 10, the surrounding final demo also includes the 10-case TP/TN/FP/FN case matrix, dashboard data, and lab-only protection action evidence.

This is not a Kibana dashboard or web UI. It is a deterministic report/dashboard command that can run without live Elasticsearch, Kafka, or Kibana.

## Pipeline

```text
implemented modules -> deterministic validation -> capability matrix -> JSON/Markdown report
```

The report reuses existing callable helpers for live telemetry, Kafka dry-run detection, SOAR dry-run response planning, ML-style anomaly scoring, and detection coverage. The Phase 9 and Phase 10 demo artifacts are generated by separate deterministic commands so FP/FN evidence and protection records remain explicit.

## CLI Usage

Generate both reports:

```powershell
python scripts\reporting\generate_final_demo_report.py
```

Write to a custom output directory:

```powershell
python scripts\reporting\generate_final_demo_report.py --output-dir reports
```

Generate JSON only:

```powershell
python scripts\reporting\generate_final_demo_report.py --format json
```

Generate Markdown only:

```powershell
python scripts\reporting\generate_final_demo_report.py --format markdown
```

Include optional Elasticsearch counts:

```powershell
python scripts\reporting\generate_final_demo_report.py --include-elasticsearch --elasticsearch-url http://localhost:9200
```

## Report Artifacts

Default artifact paths:

- `reports/final_demo_report.json`
- `reports/final_demo_report.md`

The JSON report includes `generated_at`, `project_status`, `implemented_phases`, `capability_matrix`, `validation_results`, `demo_command_checklist`, `known_limitations`, and `out_of_scope`.

When `--include-elasticsearch` is set, it also includes `elasticsearch_counts`.

## Format Options

- `--format json` writes only `final_demo_report.json`.
- `--format markdown` writes only `final_demo_report.md`.
- `--format all` writes both files and is the default.

## Optional Elasticsearch Counts

Elasticsearch is not called by default. With `--include-elasticsearch`, the command queries `_count` for:

- `edr-normalized-events-*`
- `edr-alerts-native-*`
- `edr-response-actions-*`
- `edr-protection-actions-*`

If the requested Elasticsearch query fails or returns malformed JSON, the CLI exits with code `2`.

## Validation Checks

The report validates:

- Live telemetry fixture detectable PowerShell produces a native alert.
- Live telemetry fixture detectable PowerShell produces a Sigma-like alert.
- Kafka dry-run fixture produces a native alert.
- Kafka dry-run fixture produces a Sigma-like alert.
- SOAR fixture alert produces one dry-run response record.
- ML benign fixture produces low score and no alert.
- Detection coverage report validation passes.

Phase 9 dashboard validation additionally generates:

- `reports/demo_cases/case_matrix.json`
- `reports/demo_cases/dashboard_data.json`

Those artifacts preserve true positives, true negatives, false positives, and false negatives as evaluation evidence. Do not remove FP/FN rows to make the demo look cleaner.

Phase 10 protection validation uses guarded lab-only protection records in `edr-protection-actions-*`. Kill-process remains dry-run by default and requires explicit lab flags for execution.

## Demo Command Checklist

```powershell
python scripts\smoke\end_to_end_art_telemetry_smoke.py
python scripts\detection\run_native_detection.py
python scripts\pipeline\run_live_telemetry_pipeline.py --input fixture --fixture-detectable-powershell --engine all
python scripts\kafka\produce_normalized_event.py --input fixture --fixture-detectable-powershell --dry-run
python scripts\kafka\consume_and_detect.py --dry-run-fixture --engine all
python scripts\response\run_soar_response.py --input fixture-alert
python scripts\response\run_protection_action.py --input fixture-alert --action kill-process --output summary
python scripts\ml\run_process_anomaly_detection.py --input fixture
python scripts\reporting\generate_detection_coverage_report.py
python scripts\reporting\generate_final_demo_report.py
python scripts\demo\run_demo_case_matrix.py --output reports\demo_cases\case_matrix.json
python scripts\demo\generate_demo_dashboard_data.py --case-matrix reports\demo_cases\case_matrix.json --output reports\demo_cases\dashboard_data.json
```

## Known Limitations

- Elasticsearch and Kibana are optional unless live index counts are requested.
- Kafka validation uses deterministic in-memory dry-run paths.
- SOAR remains dry-run only and does not execute containment.
- Lab-only kill-process requires explicit `--execute-protection` and `--lab-allow-execute` flags.
- ML anomaly detection is heuristic and deterministic, not trained production ML.

## Out Of Scope

- New detection semantics
- New Sysmon Event IDs
- TheHive
- Production containment
- Host isolation or network blocking
- Dashboards requiring Kibana API
- Heavy ML frameworks
