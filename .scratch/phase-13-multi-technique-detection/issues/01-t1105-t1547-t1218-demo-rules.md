# T1105, T1547.001, and T1218 demo detection rules

## Goal

Add multi-technique ATT&CK demo detection coverage using the expanded Sysmon telemetry from Phase 12.

This phase adds deterministic detection semantics for:

- `T1105` Ingress Tool Transfer through safe LOLBin download markers.
- `T1547.001` Registry Run Keys / Startup Folder persistence through Sysmon Event ID 13.
- `T1218` System Binary Proxy Execution through safe rundll32/regsvr32/mshta markers.

The result should be demo-ready, testable without live infrastructure, and safe: no malware payloads, no credential dumping, no external downloads, and no real containment.

## Context

Current project capabilities:

- Sysmon Event ID `1`, `3`, `11`, and `13` normalization.
- `normalize_sysmon_event()` router for Sysmon telemetry.
- Native and Sigma-like `T1059.001` PowerShell detection.
- ML anomaly detection.
- SOAR dry-run response.
- Lab-only kill-process protection.
- Phase 9 demo case matrix with TP/TN/FP/FN transparency.
- Detection coverage report, final demo report, and dashboard data generation.

Phase 12 added telemetry breadth. This issue uses that telemetry to broaden ATT&CK detection coverage while keeping the existing `T1059.001` behavior stable.

## Problem

The EDR currently detects PowerShell `T1059.001` and ML anomaly cases well, but ATT&CK coverage is still narrow. With normalized Event ID `3`, `11`, and `13` available, the project can show more realistic EDR coverage:

- Network and file evidence for download/transfer behavior.
- Registry value set evidence for Run key persistence.
- Process execution evidence for LOLBin proxy execution.

These rules must stay deterministic and lab-safe. They should detect explicit demo markers and constrained local indicators, not broad suspicious behavior that would create noisy false positives.

## Technical design

### Rule loading and engine integration

The existing native detector is YAML-driven for `T1059.001`, while this issue asks for Python native rule modules. Keep this expansion minimal:

- Preserve current `T1059.001` loader/evaluator behavior.
- Add a small native multi-rule registry that can evaluate the existing native YAML rule and the new Python-native rules together.
- Keep each new native rule as a pure function or small module-level rule object; no file I/O, network I/O, Windows dependency, Elasticsearch dependency, or Docker dependency.
- Add a Sigma-like multi-rule loader for files under `detection/rules/sigma_like/rules/`.
- Reuse the existing alert builder style, but extend selected context fields so alerts can carry `file`, `registry`, `destination`, and `network` data when present.

Suggested native module shape:

```python
from typing import Any

RULE = {
    "id": "det.t1105.lolbin_download",
    "version": 1,
    "name": "T1105 LOLBin download marker",
    "severity": "medium",
    "confidence": "medium",
}


def match(event: dict[str, Any]) -> tuple[bool, tuple[str, ...]]:
    """Return whether this normalized ECS event matches the deterministic demo rule."""
    ...
```

Trade-off: a Python rule module is easier for compound logic across process, network, and file events. The YAML native rule engine is simpler and already tested. Best practice here is to add the smallest adapter/registry needed for both styles and keep the alert document contract identical.

### Rule 1: T1105 LOLBin download

Native rule ID:

- `det.t1105.lolbin_download`

Sigma-like rule ID:

- `sigma_like.t1105.lolbin_download`

Match any of these deterministic paths.

Process event path:

- `event.category` contains `process`
- `process.name` is one of:
  - `certutil.exe`
  - `bitsadmin.exe`
  - `curl.exe`
  - `powershell.exe`
- `process.command_line` contains at least one safe download marker:
  - `urlcache`
  - `bitsadmin`
  - `Invoke-WebRequest`
  - `iwr`
  - `curl`
  - `http://127.0.0.1`
  - `example.test`
  - `EDR_DEMO_T1105`

Network event path:

- `event.code = 3`
- `process.name` is one of:
  - `certutil.exe`
  - `bitsadmin.exe`
  - `curl.exe`
  - `powershell.exe`
- `destination.ip = 127.0.0.1` or `destination.domain = example.test`
- Prefer requiring `EDR_DEMO_T1105` in `process.command_line` or `sysmon.event_data` when the field is available.

File create event path:

- `event.code = 11`
- `file.path` contains `Downloads`
- `file.name` contains `edr_demo`
- `sysmon.event_data` or `file.path` contains `EDR_DEMO_T1105` or `edr_demo`

ATT&CK metadata:

- Tactic: `Command and Control`
- Technique ID: `T1105`
- Technique name: `Ingress Tool Transfer`
- Severity: `medium`
- Confidence: `medium` or `high`; use `high` only when the explicit `EDR_DEMO_T1105` marker is required.

### Rule 2: T1547.001 Registry Run Key Persistence

Native rule ID:

- `det.t1547_001.registry_run_key_persistence`

Sigma-like rule ID:

- `sigma_like.t1547_001.registry_run_key_persistence`

Match:

- `event.code = 13`
- `event.category` contains `registry`
- `registry.path` contains one of:
  - `\Software\Microsoft\Windows\CurrentVersion\Run`
  - `\Software\Microsoft\Windows\CurrentVersion\RunOnce`
- `registry.value` exists
- `registry.data.strings` or `sysmon.event_data.Details` contains at least one:
  - `powershell.exe`
  - `cmd.exe`
  - `rundll32.exe`
  - `EDRDemo`
  - `EDR_DEMO_T1547`

ATT&CK metadata:

- Tactic: `Persistence`
- Technique ID: `T1547.001`
- Technique name: `Registry Run Keys / Startup Folder`
- Severity: `high`
- Confidence: `high`

### Rule 3: T1218-lite suspicious LOLBin execution

Native rule ID:

- `det.t1218.lolbin_suspicious_execution`

Sigma-like rule ID:

- `sigma_like.t1218.lolbin_suspicious_execution`

Match:

- `event.category` contains `process`
- `process.name` is one of:
  - `rundll32.exe`
  - `regsvr32.exe`
  - `mshta.exe`
- `process.command_line` contains at least one safe marker:
  - `EDR_DEMO_T1218`
  - `javascript:`
  - `scrobj.dll`
  - `url.dll,FileProtocolHandler`
  - `suspicious.dll`
  - `example.test`

ATT&CK metadata:

- Tactic: `Defense Evasion`
- Technique ID: `T1218`
- Technique name: `System Binary Proxy Execution`
- Severity: `medium`
- Confidence: `medium`

This is intentionally `T1218-lite`. It is a deterministic demo rule, not a full LOLBin behavior model.

## Alert document contract

Reuse the existing alert builder style. Alerts from both native and Sigma-like rules must include:

- `alert.kind`
- `alert.status`
- `alert.severity`
- `alert.confidence`
- `rule.id`
- `rule.name`
- `rule.description`
- `rule.version`
- `attack.tactic`
- `attack.technique.id`
- `attack.technique.name`
- `source.document_id`
- `source.index`
- `detection.matched_fields`

Sigma-like alerts must also include:

- `detection.engine = "sigma-like"`

When present on the normalized event, alerts should preserve relevant investigation context:

- `process.name`
- `process.executable`
- `process.command_line`
- `file.path`
- `file.name`
- `registry.path`
- `registry.value`
- `registry.data`
- `destination.ip`
- `destination.domain`
- `destination.port`
- `network.transport`

Best practice: copy only selected ECS context fields into alerts. Do not embed full raw XML or full original event payload unless an existing alert pattern already does that.

## Sigma-like design

Reuse the current Sigma-like loader/evaluator style. Do not implement a broad Sigma compiler.

Add only deterministic YAML rule support compatible with this project:

- Hard gates for `event.dataset` and `event.code` where appropriate.
- Existing operators such as `contains`, `equals`, and `endswith`.
- Minimal new operators only when tests prove they are needed, such as list-aware `contains` for `event.category` or nested dict text search for `sysmon.event_data`.

Trade-off: broad Sigma support would be more flexible, but it would add parser complexity and risk. For this project, deterministic demo rules are safer and easier to explain.

## Implementation notes

Keep the matching logic explicit and explainable. A reviewer should be able to point at each alert and say which safe marker triggered it.

Good rule behavior:

```python
matched, matched_fields = match(event)
assert matched is True
assert "attack.technique.id" not in matched_fields
assert event["event"]["code"] in {1, 3, 11, 13}
```

Avoid matching on ATT&CK metadata inside the input event. Detections should derive ATT&CK metadata from the rule, not trust the incoming event to label itself.
